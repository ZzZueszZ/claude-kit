# Database Migration

Best practices cho database schema migration và data migration.

---

## 1. Flyway Conventions

### File Structure
```
src/main/resources/db/migration/
├── V1.0.0__create_users_table.sql
├── V1.0.1__create_roles_table.sql
├── V1.0.2__create_user_roles_table.sql
├── V1.1.0__add_user_status_column.sql
├── V1.1.1__add_user_email_index.sql
├── R__refresh_materialized_views.sql    # Repeatable migration
└── afterMigrate.sql                     # Callback script
```

### Naming Convention

| Pattern | Mô tả | Ví dụ |
|---------|--------|-------|
| `V{version}__{description}.sql` | Versioned migration | `V1.2.0__add_orders_table.sql` |
| `R__{description}.sql` | Repeatable migration | `R__update_views.sql` |
| `U{version}__{description}.sql` | Undo migration (Flyway Teams) | `U1.2.0__drop_orders_table.sql` |

### Version Numbering
```
V{major}.{minor}.{patch}__{description}.sql

major: Breaking change / release lớn
minor: Feature mới (thêm table, thêm column)
patch: Fix nhỏ (thêm index, sửa constraint)
```

### Quy tắc bắt buộc
- **KHÔNG BAO GIỜ** sửa migration đã apply (checksum mismatch)
- **LUÔN** tạo migration mới cho mọi thay đổi
- **LUÔN** test migration trên copy của production DB trước
- Mỗi migration file thực hiện **1 logical change**
- **LUÔN** include rollback script (comment hoặc file riêng)

## 2. Migration File Template

### DDL Migration (Schema changes)
```sql
-- V1.2.0__create_orders_table.sql
-- Description: Create orders table for order management module
-- Author: dev-name
-- Date: 2024-01-15
-- Rollback: DROP TABLE orders;

CREATE TABLE orders (
    id            NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id       NUMBER        NOT NULL,
    order_number  VARCHAR2(50)  NOT NULL,
    total_amount  NUMBER(15,2)  NOT NULL,
    status        VARCHAR2(20)  DEFAULT 'PENDING' NOT NULL,
    notes         VARCHAR2(4000),
    created_at    TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    updated_at    TIMESTAMP     DEFAULT SYSTIMESTAMP NOT NULL,
    created_by    VARCHAR2(100),
    updated_by    VARCHAR2(100),

    CONSTRAINT fk_orders_user FOREIGN KEY (user_id) REFERENCES users(id),
    CONSTRAINT uq_orders_number UNIQUE (order_number),
    CONSTRAINT chk_orders_amount CHECK (total_amount >= 0),
    CONSTRAINT chk_orders_status CHECK (status IN ('PENDING','CONFIRMED','SHIPPED','COMPLETED','CANCELLED'))
);

-- Indexes
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_orders_status ON orders(status);
CREATE INDEX idx_orders_created_at ON orders(created_at);

-- Comments
COMMENT ON TABLE orders IS 'Customer orders';
COMMENT ON COLUMN orders.status IS 'Order status: PENDING, CONFIRMED, SHIPPED, COMPLETED, CANCELLED';
```

### DML Migration (Data changes)
```sql
-- V1.2.1__seed_order_statuses.sql
-- Description: Insert initial order status reference data
-- Author: dev-name
-- Rollback: DELETE FROM order_statuses;

INSERT ALL
    INTO order_statuses (code, name, description, sort_order) VALUES ('PENDING', 'Pending', 'Order placed, awaiting confirmation', 1)
    INTO order_statuses (code, name, description, sort_order) VALUES ('CONFIRMED', 'Confirmed', 'Order confirmed by admin', 2)
    INTO order_statuses (code, name, description, sort_order) VALUES ('SHIPPED', 'Shipped', 'Order shipped to customer', 3)
    INTO order_statuses (code, name, description, sort_order) VALUES ('COMPLETED', 'Completed', 'Order delivered successfully', 4)
    INTO order_statuses (code, name, description, sort_order) VALUES ('CANCELLED', 'Cancelled', 'Order cancelled', 5)
SELECT 1 FROM DUAL;
```

## 3. Safe Migration Patterns

### ✅ Backwards-Compatible Changes (Safe)

| Operation | Cách làm |
|-----------|---------|
| Thêm table mới | `CREATE TABLE` — an toàn |
| Thêm column nullable | `ALTER TABLE ADD column` — an toàn |
| Thêm column có default | `ALTER TABLE ADD column DEFAULT value` — an toàn Oracle 12c+ |
| Thêm index | `CREATE INDEX` — an toàn (dùng ONLINE nếu có thể) |
| Thêm constraint | Kiểm tra data trước, dùng `NOVALIDATE` nếu cần |

### ⚠️ Dangerous Operations

#### Rename Column (2 migrations)
```sql
-- Migration 1: Thêm column mới + copy data
-- V1.3.0__add_full_name_column.sql
ALTER TABLE users ADD full_name VARCHAR2(200);
UPDATE users SET full_name = name;
ALTER TABLE users MODIFY full_name NOT NULL;

-- Deploy code hỗ trợ CẢ 2 columns (đọc full_name, ghi cả 2)

-- Migration 2: Xóa column cũ (sau khi code đã migrate xong)
-- V1.4.0__drop_name_column.sql
ALTER TABLE users DROP COLUMN name;
```

#### Change Column Type
```sql
-- ⚠️ KHÔNG: ALTER TABLE users MODIFY email VARCHAR2(500);
-- (Có thể fail nếu data không compatible)

-- ✅ Safe approach:
-- 1. Thêm column mới
ALTER TABLE users ADD email_new VARCHAR2(500);
-- 2. Copy data
UPDATE users SET email_new = email;
-- 3. Swap (sau khi code migrate)
ALTER TABLE users DROP COLUMN email;
ALTER TABLE users RENAME COLUMN email_new TO email;
```

#### Drop Column
```sql
-- ⚠️ KHÔNG drop column ngay — code cũ có thể đang reference
-- Multi-step approach:

-- Step 1: Code ignore column (không SELECT, không INSERT)
-- Step 2: Mark column unused (nhanh, không lock)
ALTER TABLE users SET UNUSED COLUMN legacy_field;
-- Step 3: Drop unused columns (khi maintenance window)
ALTER TABLE users DROP UNUSED COLUMNS;
```

#### Drop Table
```sql
-- ⚠️ Multi-step approach:
-- Step 1: Rename table (soft drop)
ALTER TABLE old_features RENAME TO old_features_deprecated;
-- Step 2: Verify no code references
-- Step 3: Drop table (after grace period)
DROP TABLE old_features_deprecated PURGE;
```

## 4. Rollback Strategies

### Strategy 1: Forward-Fix (Recommended)
```sql
-- Luôn tạo migration MỚI để fix, không rollback
-- V1.2.0__create_feature_x.sql (có bug)
-- V1.2.1__fix_feature_x.sql   (fix bằng migration mới)
```

### Strategy 2: Manual Rollback Script
```sql
-- Mỗi migration kèm rollback script trong comment header
-- V1.2.0__create_orders_table.sql
-- Rollback:
--   DROP INDEX idx_orders_user_id;
--   DROP INDEX idx_orders_status;
--   DROP TABLE orders PURGE;
```

### Strategy 3: Backup trước Migration
```sql
-- Tạo backup table trước khi alter
CREATE TABLE users_backup_20240115 AS SELECT * FROM users;

-- Thực hiện migration...
ALTER TABLE users ADD new_column VARCHAR2(100);

-- Nếu cần rollback:
-- DROP TABLE users;
-- ALTER TABLE users_backup_20240115 RENAME TO users;
```

### Quy tắc Rollback
- **LUÔN** có rollback plan trước khi apply migration
- **ƯU TIÊN** forward-fix hơn rollback
- **LUÔN** backup data trước destructive migrations
- Test rollback script trên non-production environment

## 5. Spring Boot + Flyway Configuration

### application.yml
```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
    baseline-on-migrate: true          # Cho existing databases
    baseline-version: '0'
    validate-on-migrate: true
    out-of-order: false                # Strict ordering
    clean-disabled: true               # KHÔNG cho phép clean trong production
    table: flyway_schema_history
    schemas: ${DB_SCHEMA:MY_SCHEMA}
```

### Profile-specific
```yaml
# application-dev.yml
spring:
  flyway:
    clean-disabled: false              # Cho phép clean trong dev
    out-of-order: true                 # Linh hoạt hơn trong dev

# application-prod.yml
spring:
  flyway:
    clean-disabled: true               # TUYỆT ĐỐI không clean
    out-of-order: false
```

## 6. Oracle-specific Migration Tips

### Online Operations
```sql
-- Online index creation (không lock table)
CREATE INDEX idx_orders_status ON orders(status) ONLINE;

-- Online table redefinition (thay đổi structure không downtime)
BEGIN
    DBMS_REDEFINITION.START_REDEF_TABLE(
        uname     => 'MY_SCHEMA',
        orig_table => 'LARGE_TABLE',
        int_table  => 'LARGE_TABLE_NEW'
    );
END;
/
```

### Large Table Alterations
```sql
-- ✅ Thêm column có DEFAULT — Oracle 12c+ metadata-only (instant)
ALTER TABLE large_table ADD new_col VARCHAR2(50) DEFAULT 'N/A';

-- ⚠️ Thêm NOT NULL constraint — cần validate existing data
ALTER TABLE large_table ADD new_col VARCHAR2(50) DEFAULT 'N/A' NOT NULL;
-- Oracle 12c+: metadata-only nếu có DEFAULT

-- ✅ Add constraint NOVALIDATE (không check existing data)
ALTER TABLE large_table ADD CONSTRAINT chk_new
    CHECK (new_col IN ('A','B','C')) NOVALIDATE;
-- Validate sau (non-blocking)
ALTER TABLE large_table MODIFY CONSTRAINT chk_new VALIDATE;
```

## Checklist Migration

- [ ] File name đúng convention: `V{version}__{description}.sql`
- [ ] Mỗi file thực hiện 1 logical change
- [ ] Header comment có description, author, date, rollback
- [ ] Test trên copy của production DB
- [ ] Rollback plan documented
- [ ] Backwards-compatible (không break running code)
- [ ] FK columns có index
- [ ] Dangerous ops dùng multi-step approach
- [ ] `clean-disabled: true` trong production config
- [ ] Large table operations dùng ONLINE khi có thể
