# Database Design

Chuẩn hóa thiết kế database cho ứng dụng enterprise.

---

## 1. Naming Conventions

### Quy tắc bắt buộc
- **Table name**: `snake_case`, số nhiều → `users`, `order_items`, `user_roles`
- **Column name**: `snake_case` → `first_name`, `created_at`, `is_active`
- **Primary Key**: `id` (hoặc `{table_singular}_id` nếu cần rõ ràng)
- **Foreign Key**: `{referenced_table_singular}_id` → `user_id`, `order_id`
- **Index**: `idx_{table}_{columns}` → `idx_users_email`, `idx_orders_status_created_at`
- **Unique Constraint**: `uq_{table}_{columns}` → `uq_users_email`
- **Check Constraint**: `chk_{table}_{description}` → `chk_orders_amount_positive`
- **Sequence**: `seq_{table}` → `seq_users`, `seq_orders`

### ❌ Sai
```sql
-- ❌ camelCase
CREATE TABLE userOrders (orderId NUMBER, userId NUMBER);

-- ❌ Tên viết tắt khó hiểu
CREATE TABLE usr_ord (ord_id NUMBER, usr_id NUMBER);

-- ❌ Prefix không cần thiết
CREATE TABLE tbl_users (col_id NUMBER, col_name VARCHAR2(100));
```

### ✅ Đúng
```sql
CREATE TABLE user_orders (
    id           NUMBER GENERATED BY DEFAULT AS IDENTITY,
    user_id      NUMBER       NOT NULL,
    order_date   DATE         NOT NULL,
    total_amount NUMBER(15,2) NOT NULL,
    status       VARCHAR2(20) DEFAULT 'PENDING' NOT NULL,
    created_at   TIMESTAMP    DEFAULT SYSTIMESTAMP NOT NULL,
    updated_at   TIMESTAMP    DEFAULT SYSTIMESTAMP NOT NULL
);
```

## 2. Normalization

### Các cấp độ normalization

| Normal Form | Quy tắc | Ví dụ vi phạm |
|------------|---------|---------------|
| **1NF** | Mỗi cell chứa 1 giá trị atomic | `phones = '0901,0902'` |
| **2NF** | 1NF + không phụ thuộc vào một phần PK | Composite PK nhưng column chỉ phụ thuộc 1 phần |
| **3NF** | 2NF + không phụ thuộc bắc cầu | `zip_code → city` (city phụ thuộc zip, không phụ thuộc PK) |

### Quy tắc
- **LUÔN** đạt ít nhất **3NF** cho OLTP systems
- **CÓ THỂ** denormalize cho reporting/analytics (xem Section 6)
- **KHÔNG** lưu trữ dữ liệu tính toán được trừ khi performance yêu cầu

### ❌ Vi phạm 1NF
```sql
-- ❌ Nhiều giá trị trong 1 cột
CREATE TABLE users (
    id    NUMBER PRIMARY KEY,
    name  VARCHAR2(100),
    phones VARCHAR2(500)  -- '0901234567,0902345678'  ❌
);
```

### ✅ Đúng — Tách bảng
```sql
CREATE TABLE users (
    id   NUMBER PRIMARY KEY,
    name VARCHAR2(100) NOT NULL
);

CREATE TABLE user_phones (
    id       NUMBER PRIMARY KEY,
    user_id  NUMBER NOT NULL REFERENCES users(id),
    phone    VARCHAR2(20) NOT NULL,
    type     VARCHAR2(10) DEFAULT 'MOBILE' NOT NULL
);
```

## 3. Data Types — Chọn đúng loại

### Quy tắc bắt buộc
- **LUÔN** dùng data type nhỏ nhất phù hợp
- **KHÔNG** dùng `VARCHAR2(4000)` khi chỉ cần `VARCHAR2(100)`
- **LUÔN** dùng `NUMBER(p,s)` với precision rõ ràng cho tiền tệ
- **KHÔNG** dùng floating-point (`FLOAT`, `DOUBLE`) cho tiền tệ

### Bảng chọn Data Type

| Loại dữ liệu | Data Type | Ví dụ |
|--------------|-----------|-------|
| ID / số nguyên | `NUMBER(19)` hoặc `IDENTITY` | `id`, `quantity` |
| Tiền tệ | `NUMBER(15,2)` hoặc `NUMBER(19,4)` | `price`, `total_amount` |
| Phần trăm | `NUMBER(5,2)` | `tax_rate`, `discount_percent` |
| Chuỗi ngắn (≤ 100) | `VARCHAR2(n)` | `name`, `email`, `phone` |
| Chuỗi dài | `VARCHAR2(4000)` hoặc `CLOB` | `description`, `content` |
| Boolean | `NUMBER(1)` + CHECK | `is_active`, `is_deleted` |
| Ngày | `DATE` | `birth_date`, `order_date` |
| Ngày + giờ | `TIMESTAMP` | `created_at`, `updated_at` |
| Ngày + timezone | `TIMESTAMP WITH TIME ZONE` | `event_time` |
| Enum / Status | `VARCHAR2(20)` | `status`, `type`, `role` |
| JSON data | `CLOB` + IS JSON check | `metadata`, `config` |
| Binary | `BLOB` | `avatar`, `document` |

### ✅ Ví dụ
```sql
CREATE TABLE products (
    id          NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    sku         VARCHAR2(50)    NOT NULL,
    name        VARCHAR2(200)   NOT NULL,
    description VARCHAR2(4000),
    price       NUMBER(15,2)    NOT NULL,
    quantity    NUMBER(10)      DEFAULT 0 NOT NULL,
    weight_kg   NUMBER(8,3),
    is_active   NUMBER(1)       DEFAULT 1 NOT NULL,
    metadata    CLOB            CHECK (metadata IS JSON),
    created_at  TIMESTAMP       DEFAULT SYSTIMESTAMP NOT NULL,

    CONSTRAINT chk_products_price_positive CHECK (price > 0),
    CONSTRAINT chk_products_quantity_non_negative CHECK (quantity >= 0),
    CONSTRAINT chk_products_is_active CHECK (is_active IN (0, 1)),
    CONSTRAINT uq_products_sku UNIQUE (sku)
);
```

## 4. Primary Key & Foreign Key

### Quy tắc bắt buộc
- **LUÔN** có Primary Key cho mỗi bảng
- **ƯU TIÊN** surrogate key (`id` auto-generated) hơn natural key
- **LUÔN** tạo Foreign Key constraints cho data integrity
- **LUÔN** tạo index trên Foreign Key columns (Oracle **KHÔNG** tự tạo)

### Surrogate Key vs Natural Key

| | Surrogate Key | Natural Key |
|---|---|---|
| **Ưu điểm** | Ổn định, không thay đổi | Có ý nghĩa business |
| **Nhược điểm** | Không có ý nghĩa business | Có thể thay đổi, composite phức tạp |
| **Khi nào dùng** | Hầu hết bảng | Config tables, lookup tables |

### Foreign Key với ON DELETE
```sql
-- CASCADE: Xóa parent → xóa children
ALTER TABLE order_items
    ADD CONSTRAINT fk_order_items_order
    FOREIGN KEY (order_id) REFERENCES orders(id)
    ON DELETE CASCADE;

-- SET NULL: Xóa parent → set null children
ALTER TABLE employees
    ADD CONSTRAINT fk_employees_manager
    FOREIGN KEY (manager_id) REFERENCES employees(id)
    ON DELETE SET NULL;

-- RESTRICT (default): Không cho xóa parent nếu có children
ALTER TABLE orders
    ADD CONSTRAINT fk_orders_user
    FOREIGN KEY (user_id) REFERENCES users(id);
    -- Mặc định là RESTRICT
```

### ⚠️ Lưu ý Oracle
```sql
-- Oracle KHÔNG tự tạo index cho FK → PHẢI tạo thủ công
CREATE INDEX idx_order_items_order_id ON order_items(order_id);
CREATE INDEX idx_orders_user_id ON orders(user_id);
```

## 5. Indexing Strategy

### Quy tắc bắt buộc
- **LUÔN** tạo index cho Foreign Key columns
- **LUÔN** tạo index cho columns dùng trong WHERE, JOIN, ORDER BY thường xuyên
- **KHÔNG** tạo index cho bảng nhỏ (< 1000 rows)
- **KHÔNG** over-index — mỗi index chậm INSERT/UPDATE/DELETE

### Khi nào tạo Index

| Trường hợp | Loại Index | Ví dụ |
|-----------|-----------|-------|
| Lookup bằng equality | B-tree (default) | `WHERE email = ?` |
| Tìm kiếm range | B-tree | `WHERE created_at > ?` |
| Cột có ít distinct values | Bitmap (Oracle) | `WHERE status = 'ACTIVE'` |
| Tìm kiếm nhiều cột | Composite | `WHERE status = ? AND type = ?` |
| Tìm kiếm case-insensitive | Function-based | `WHERE UPPER(email) = ?` |

### Composite Index — Thứ tự cột quan trọng
```sql
-- Composite index: cột có selectivity CAO đặt TRƯỚC
-- Thứ tự: equality columns trước, range columns sau
CREATE INDEX idx_orders_status_created
    ON orders(status, created_at);

-- ✅ Dùng được index
SELECT * FROM orders WHERE status = 'PENDING' AND created_at > SYSDATE - 7;
SELECT * FROM orders WHERE status = 'PENDING';

-- ❌ KHÔNG dùng được index (skip leading column)
SELECT * FROM orders WHERE created_at > SYSDATE - 7;
```

### Function-based Index
```sql
-- Cho case-insensitive search
CREATE INDEX idx_users_email_upper ON users(UPPER(email));

-- Query phải match function
SELECT * FROM users WHERE UPPER(email) = UPPER('user@example.com');
```

## 6. Denormalization

### Khi nào denormalize
- **CHỈ KHI** đã chứng minh bottleneck qua query profiling
- Reporting tables / materialized views
- Caching computed values cần truy xuất thường xuyên

### Patterns an toàn
```sql
-- 1. Computed column (trigger-maintained)
ALTER TABLE orders ADD total_items NUMBER(10) DEFAULT 0;
-- Cập nhật bằng trigger khi order_items thay đổi

-- 2. Materialized View cho reporting
CREATE MATERIALIZED VIEW mv_monthly_sales
REFRESH FAST ON DEMAND
AS
SELECT
    TRUNC(order_date, 'MM') AS month,
    COUNT(*)                AS order_count,
    SUM(total_amount)       AS total_revenue
FROM orders
WHERE status = 'COMPLETED'
GROUP BY TRUNC(order_date, 'MM');

-- 3. Summary table cho dashboard
CREATE TABLE daily_statistics (
    stat_date    DATE PRIMARY KEY,
    new_users    NUMBER(10) DEFAULT 0,
    new_orders   NUMBER(10) DEFAULT 0,
    total_revenue NUMBER(15,2) DEFAULT 0
);
```

### Quy tắc
- **LUÔN** document lý do denormalize
- **LUÔN** có cơ chế sync data (trigger, scheduled job, event)
- **KHÔNG** denormalize premature (chưa có evidence)

## Checklist thiết kế database

- [ ] Tên table/column dùng `snake_case`
- [ ] Đạt ít nhất 3NF (trừ khi có lý do denormalize rõ ràng)
- [ ] Data types phù hợp (NUMBER có precision, VARCHAR2 có length)
- [ ] Mọi bảng có Primary Key
- [ ] Foreign Key constraints đầy đủ
- [ ] Index trên FK columns (Oracle không tự tạo)
- [ ] Index trên columns dùng WHERE/JOIN/ORDER BY thường xuyên
- [ ] NOT NULL cho required columns
- [ ] DEFAULT values cho columns có giá trị mặc định
- [ ] CHECK constraints cho data validation
- [ ] UNIQUE constraints cho business-unique columns
- [ ] Không dùng FLOAT/DOUBLE cho tiền tệ
